# Spark `.spark/` store inventory

This inventory is the reviewable contract for local Spark runtime state under a project
root. It documents the current owners and guarantees; it does not require every store to
have the same durability level.

## Scope classes

- **Project canonical/protected** stores are project-level Spark state. They are not safe
  housekeeping targets except through a typed owner API or a future explicit cleanup
  command with user intent.
- **Session cache** stores live under the project `.spark/` directory but are scoped to a
  Pi session, owner session, or display cache. They may be stale after sessions finish and
  are eligible for housekeeping when the owning workflow is terminal or the cache is
  broken/stale.
- **Legacy/import-only** stores are read for migration or compatibility only. New writes
  must go to the current owner API.
- **Ad hoc scratch** paths are human/agent files under `.spark/` without typed schemas.
  Prefer typed artifacts for durable workflow evidence.

## Global transaction rules

- `.spark/` is local runtime state. It should be ignored by Git; durable shared knowledge
  must be exported or committed explicitly outside `.spark/`.
- Each typed store has one owner package/API. Callers should go through that API instead
  of writing JSON files directly. Where the current implementation has no dedicated store
  API, that gap is called out below rather than hidden.
- Store transaction boundaries are per store file or per artifact metadata file. Spark does
  **not** provide database-style cross-store transactions.
- Multi-store flows must tolerate partial success. For example, task execution may update
  `thread.json` and create artifacts separately; `/spark` initialization creates graph,
  TODO, review gate, and artifacts in separate commits.
- When a store writes content-addressed blobs plus metadata, metadata is the commit point.
  Orphan blobs are acceptable and may be reported by housekeeping; they are not evidence
  until referenced by committed metadata.
- Session caches should be treated as recoverable hints. Losing or deleting a cache file
  must not delete canonical thread/task/artifact history.

## Owner boundaries by package

- `spark-tasks` owns the `TaskGraph` schema and the task TODO store schema. It must not
  execute roles or write Spark artifacts.
- `spark-artifacts` owns physical artifact metadata and blobs. Producers own the meaning
  of their artifact `kind` and body, but not the filesystem layout.
- `spark-learnings`, `spark-ask`, `spark-runtime`, and review flows own logical artifact
  bodies while delegating persistence to `spark-artifacts`.
- `spark-orchestrator` owns DAG-manager invocation state in `dag-runs.json`; it does not
  own task graph snapshots.
- `spark-review` owns review gate/artifact semantics. The current `review-gate.json` file
  writer still lives in the `spark` facade and should remain a known boundary gap until a
  dedicated store is introduced.
- `spark` owns Pi extension/session caches (`current-thread`, `session-todos`,
  `todo-display-numbers`), the prototype roadmap planning layer (`roadmap.json`), and the
  housekeeping summary that classifies cache vs protected state.
- `pi-roles` does not own Spark workflow state. Its only `.spark/` path is legacy
  read-only role import from `.spark/agents`.

## Housekeeping strategy

Housekeeping must classify before deleting:

1. **Protected project state:** `thread.json`, `artifacts/`, `dag-runs.json`,
   `review-gate.json`, `roadmap.json`, and `notes/` are report-only unless a typed owner cleanup API says
   otherwise.
2. **Session caches:** `current-thread/`, `todos/<session>.json`, `session-todos/`, and
   `todo-display-numbers/` may be reported as safe-to-delete when stale, broken, empty, or
   tied only to terminal/missing work according to the cache-specific rules above.
3. **Legacy state:** `todos.json` and `agents/*.json` should be migrated or explicitly
   retired; do not write new canonical state there.
4. **Scratch:** `role-reports/` and untyped files are not canonical evidence. Cleanup
   should be conservative and prefer reporting before deletion.

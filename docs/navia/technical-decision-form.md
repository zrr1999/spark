# Standalone Navia technical decision form

Date: 2026-05-20

## Scope

This form is for the standalone **Navia** product/project in this workspace. It is not `pi-spark` and not a Pi extension. pi-spark/Pi concepts may be handled later through explicit adapters, but they are not product architecture defaults here.

Architecture pivot: Navia should now follow a lightweight Multica-like split — **SvelteKit frontend/server + SQLite projections + external runtime boundary**. The SvelteKit server primarily owns communication-plane concerns: auth, sessions, workspace bindings, command/human-response delivery records, projections, lazy artifact cache, events/audit, and web fanout. The external runtime owns local workspace/CLI execution, task graph truth, orchestration, logs/checkpoints, canonical artifacts, workdirs/checkouts, and safety policy. This supersedes the earlier pure local filesystem-watch/narrow-backend-only direction, the earlier thin-runner/server-scheduler framing, and the earlier Hono/PostgreSQL-first backend baseline. Elm-style UI can still be used as a frontend internal pattern.

Repository boundary update: this Navia repo should focus on **SvelteKit frontend/server, protocol/contracts, SQLite-backed projections, lazy artifact cache/proxy, connection diagnostics, and UI**. Runtime implementation should live in a separate repository.

Use this as the single decision form before writing RFCs or implementation tasks.

## How to answer

This form should be collected as a **multi-question decision flow**, not as one giant single-select question. Each decision ID below is its own question or a small grouped section.

- If the defaults are acceptable, answer: `accept-defaults`.
- If you want changes, answer with decision IDs and chosen option IDs, for example:

```text
D01=workspace-first
D03=sqlite-db-with-exportable-files
D06=sveltekit-fullstack
D16=runner-owned-task-graph-projections
```

- If an option is missing, use `other:<short note>` for that decision ID.
- Human decision collection should have no timeout; unanswered decisions stay pending.

## Ask/form UX requirement

Do **not** render this as one question with dozens of options. Do **not** fake a form by creating combined option permutations. Use a true form ask with multiple questions and multiple options per question, grouped by section:

1. Product shape: D01-D03
2. Implementation baseline: D04-D06
3. UI architecture: D07-D08
4. Backend/runtime: D09-D10
5. Execution safety: D11-D12
6. Collaboration/evidence: D13-D18
7. Quality/release workflow: D19-D20
8. Integration/deployment/language: D21-D23

If only a single-question ask tool is available, grouped technical decision collection is blocked; use/implement an ask_flow-style form surface instead of degrading the UX into a large option list or combination list.

## Implementation detail decisions

The high-level D01-D23 decisions are now complemented by selected v0.1 implementation defaults in [`docs/rfcs/implementation-options-rfc.md`](./docs/rfcs/implementation-options-rfc.md): pnpm monorepo, Node 26, SvelteKit adapter-node, custom Node server for runtime WebSocket, native `node:sqlite`, Kysely adapter, explicit SQL migrations, Zod protocol package, local owner setup, runtime enrollment token auth, SSE browser events, in-process background jobs, lazy artifact cache, and no direct diagnostic/pairing implementation.
